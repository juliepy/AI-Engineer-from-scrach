# Agent Harness、Loop Engineering 与 LLMOps

> 来源：Shawn 视频文稿（Harness 控马、循环工程、评测与运维闭环）  
> 前置：可先看 Memory 三柱（Working / Procedural / Semantic / Episodic）

---

## 一、核心结论（先记住这句）

**LLM 是一匹很强的马；Harness 是缰绳与鞍具。**  
单次问答不够用——要用 Memory + Tool Loop + 结束护栏把模型「控住」；再用 Trace → Eval → Diagnose → Ship 的 LLMOps 闭环，让系统持续变好。

热词并不玄：Harness、Loop Engineering、LLMOps、Eval 都是**简单积木**，拼起来才是能跑的智能系统。

---

## 二、从一次 Agent Run 说起

一次 **Agent Run** = 用户输入 →（可选多轮工具调用）→ Agent 回复。

最小形态（无长期记忆、短暂 ephemeral）：

```
User Prompt + Chat History + System Prompt
        │
        ▼
  Working Memory / Context RAM
        │
        ▼
     LLM（Q&A Agent）
        │
        ▼
      Reply
```

问题：只有短时上下文。模型「懂人类知识」，但**不懂你、不懂你的软件要它怎么跑**——所以需要 Harness。

---

## 三、什么是 Harness（缰绳）

### 字面意思

骑马时的一套控制工具。LLM 像马：力大、能跑，但没有缰绳会乱跑、伤人；战争里更不能失控。

### 工程含义

**Harness = 围绕 LLM 搭的控制框架**，让它按你想要的方式工作，压住「下一词概率」带来的随机性。

市场上常见积木：LangGraph、LangChain、Pydantic AI 等（本篇讲概念，不深挖某一框架）。

### Harness 里通常有什么

| 模块 | 作用 |
|------|------|
| Working Memory | 本轮上下文组装 |
| 三柱 Memory + DB + 更新机制 | 让记忆可存、可取、可演进 |
| Tool Calling + Loop | 多步任务，直到「够好」再回复 |
| End Loop Guardrails | 何时停、何时问人 |
| （外圈）LLMOps | 可观测、可评测、可迭代 |

---

## 四、Harness 视角下的 Memory（速览）

LLM 自己做不到「记得你」——Memory 是 Harness 的一部分。

| 类型 | 存什么 | 存储 / 召回要点 |
|------|--------|-----------------|
| **Procedural** | 怎么表现、技能指令 | 多为文件 / 文本（Skills = Markdown）；也可进库 |
| **Semantic** | 耐久事实、画像 | 向量 + **RAG**；可人工写入，也可从情节巩固 |
| **Episodic** | 带时间戳的事件 / 历史会话 | 时间序列；可 SQL 查近期，也可 RAG 语义筛相关 |

### 更新与门控（Harness 要「聪明」）

```
Agent 回复 → 写入 DB（情节日志）
                │
         Gate（例：每 2000 次会话再巩固）
                │
                ▼
        Summarizer Agent（可换便宜模型）
                │
                ▼
        Distill → Semantic Memory（耐久事实）
```

电商例：百万次「怎么退款」对话 → 门控后摘要成事实，而不是每次全量扫百万日志。

### 情节检索：不只是 RAG

| 问法 | 更合适的取法 |
|------|----------------|
| 「和这位美国客户最近 10 次对话」 | 偏 **SQL / 时间过滤** |
| 「过去 20 次产品质量投诉且未解决」 | **SQL（时间/表）+ RAG（语义）**，只要相关 20 条，不要整段 2000 条 |

Semantic：事实文本 → 主要靠 RAG。  
Episodic：时间线 + 文本 → 结构化查询与语义检索常一起用。

## 五、Loop Engineering（循环工程）

### 为什么需要 Loop

Agent 不只读记忆，还会调工具，且往往**多轮**：

- 约会议
- 读写 CRM
- 从 Stripe / 支付宝拉支付、触发退款

若给 LLM「全权」：可能无限调工具、不知道何时停、不知道何时该回复用户。

### Loop 是什么

**架构层思考：怎样算「够好」，可以停下来给用户 / 业务方一个答复。**

例：用户问——客户在抱怨什么？怎么挽回？要退款的退了没有？没退能否退？

可能循环：

1. 调 CRM → 近 2 月 30 条投诉，12 已退、8 未退  
2. 再调工具 → 给未退的 8 人约跟进会  
3. 更进一步 → 直接触发 Stripe / 支付宝退款  

直到任务完成（或到达约定终点）→ **End Loop Guardrails** 触发 → 回复用户。

### End Loop Guardrails（结束护栏）

| 做法 | 说明 |
|------|------|
| 任务完成即停 | 最简护栏 |
| 规划时与用户确认终点 | 「直接退那 8 人」vs「只列出名单你自己跟」→ 两种不同结束条件 |
| 卡住时通知人 | 如 Claude Code：权限弹窗时通知笔记本，避免挂 25 分钟你还在刷 YouTube |

**没有万能 Loop**：终点取决于任务与系统设计；Harness 的关键是**显式定义停条件**。

## 六、整体 Harness 一图

```
                    ┌──────── Harness ────────┐
User Prompt ──► Working Memory ◄── Memory DB（P/S/E + RAG/SQL）
                    │
                    ▼
              LLM Agent ◄──► Tools（CRM / 支付 / 日程…）
                    │              ▲
                    │         Loop + Guardrails
                    ▼              │
                 Reply ────────────┘
                    │
                    └──► 写入 Episodic / 门控巩固 Semantic
```

到这里：Memory + Loop + 停条件 = **一套控住 LLM 的 Harness**。

## 七、为什么还要 LLMOps / Eval

Harness 能跑，但**不知道跑得好不好**。  
需要反馈闭环：评测 → 诊断 → 修（Prompt / 模型配置 / 检索参数 / 工具）→ 再跑。

这就是 **LLMOps（Large Language Model Operations）**；其中 **Eval** 是「好不好、健不健康」的度量。

YC 常提的 Eval / LLMOps，本质是：**可观测 + 可评测 + 可迭代**。

---

## 八、LLMOps 闭环

### 1. Trace（追踪）——先采数据

每个 Agent Run 记一棵**事件树**（工具：LangFuse、LangSmith 等）：

- 用户到底问了什么  
- 检索了哪些记忆  
- 工具调了几次、用得怎样  
- 端到端延迟、Token 消耗  

**Trace = 收集运行事实**，供后面两件事用：好不好？健不健康？

### 2. Eval（评测）——好不好 / 健不健康

| 维度 | 例子 |
|------|------|
| 任务是否完成 | 约会议是否真触发 |
| 延迟 | 20s vs 2ms |
| Token / 成本 | 本轮检索 + 工具调用花了多少 |

可用确定性代码，也可用 **LLM-as-Judge** 打分。

### 3. Diagnose（诊断）——哪里坏、为什么

例：

- 会议从未触发 → 喂给 Coding Agent 深挖  
- 延迟 20s → 某工具慢？Working Memory 太大？  
- 简单问题（生日、OpenAI 何时成立）却狂检索 → 检索策略过重  

要有 **Dashboard**：指标可见，才能定位旋钮。

### 4. Gate + Ship（门控发布）

```
Trace → Eval → Diagnose
              │
         评测通过？
         ├── 是：小改即发（新 Prompt / 模型配置 / 工具 / 检索参数）
         │         └──► 回灌 Harness 的 Agent Run
         └── 否（深坑）：修 Bug → 重跑 → 再 Trace → 再 Eval
```

一轮 LLMOps = 观测 → 评测 → 诊断 → 把更好的 Prompt / 配置喂回 Harness。

---

## 九、Zoom Out：两层系统

| 层 | 管什么 |
|----|--------|
| **Harness（内圈）** | 单次 / 多步怎么跑：Memory、Tool Loop、停条件、回复 |
| **LLMOps（外圈）** | 跑得怎样：Trace、Eval、诊断、改旋钮、再发布 |

目标形态：**受控执行 + 自我演进**——不是一次写死 Prompt，而是持续根据运行数据变好。

---

## 十、面试口述版（可直接背）

1. **Harness**：把 LLM 当马，用框架控随机性——Memory、工具、循环、停条件都是缰绳。  
2. **Memory**：Working 装本轮；Procedural 管怎么做；Semantic 管耐久事实（RAG）；Episodic 管时间线（SQL + RAG）；门控摘要防库爆炸。  
3. **Loop Engineering**：多工具调用直到任务够好；**End Loop Guardrails** 定义停点，必要时问人或弹通知。  
4. **LLMOps**：每个 Run 做 Trace → Eval（好不好/健不健康）→ Diagnose → 改 Prompt/配置/检索 → 回灌；深坑则修 Bug 再跑。  

一句话：**Harness 让它按你的规矩跑；LLMOps 让你知道跑得怎样并持续改规矩。**
