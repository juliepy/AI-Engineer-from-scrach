# Hermes Agent Harness 与循环工程（完整笔记）

> 来源：Shawn 视频文稿（Nous Research / Hermes Agent 实机走读）  
> 前置：通用 Harness、Loop Engineering、三柱 Memory、LLMOps（见 `08` / `10`）

> 定位：开源、本地、自改进 Agent；GitHub 短时间 20 万+ Star

---

## 一、核心结论（先记住这句）

**Hermes = 跑在本地机器上的标准 Agent Harness，外加「经验→技能/记忆」的自改进环。**  
和 Claude Code / OpenClaw 很像（终端、浏览器、子 Agent、Cron、Gateway），差异主要在：

1. **完全本地**：Skills / Memory / 会话状态不落云  
2. **内置学习环**：从错误与重复任务里自己写技能、自己沉淀记忆（不只靠人显式「请做 skill」）  
3. **Semantic 用纯文本 + 关键词**，不是 Embedding / RAG  
4. **缺完整 LLMOps / Eval**（有 Trace/Logs，无评测闭环）——要自己补

热词（Harness、Loop、Memory、Gateway、Eval）在 Hermes 里都能对上号；Hermes 是「把上一期架构图画成产品」的好例子。

## 五、Hermes Harness 基础骨架

### 5.1 一次 Agent Run（短暂 ephemeral）

```
用户指令（Desktop 或 WhatsApp Gateway）
        │
        ▼
 User Prompt + Chat History + System Prompt（SOUL.md）
        │
        ▼
   Working Memory / Context RAM
        │
        ▼
   Hermes LLM Agent（演示用 Gemini）
        │
        ▼
   Loop：调用 Hermes 侧工具（终端 / 浏览器 / Delegate / Cron / Skills / MCP…）
        │
        ▼
   End Loop Guardrails → 回复用户
```

和通用故事一样：单次 Run 本身短暂；长期能力靠 Memory + Skills 回灌 Working Memory。

### 5.2 Harness 再提醒一句

Harness = 控住「强力的马（LLM）」的一套工具与约束，别神化——热词而已。  
马 = LLM；马具 = Working Memory + Tools + Loop 停条件 + Memory 更新。

## 六、System Prompt：`SOUL.md`（「灵魂」）

### 在哪

本机 **`.Hermes`** 文件夹 → **`SOUL.md`**（桌面侧栏可打开本地文件编辑、保存）。

### 演示改了什么（皮卡丘）

- 每条回复以 `pika pika pika pi` **开头和结尾**
- 兴奋：说 `pikapi` + 闪电表情
- 悲伤：小声说 `pika`

### 和 Claude Code 对照

Claude Code 类似能力在 **Customize → General** 的 instruction：告诉 Claude 要做什么 / 不要做什么 = 贯穿整个 Agent Harness 的 System Prompt。  
Hermes 只是把这块命名成更有感觉的 **Soul**。

## 七、Loop Engineering：Hermes 有哪些工具

目标：少「逐步口头指挥」，多让 Agent **自己决定**该问什么、该调什么工具，直到任务完成再回复。

| 工具 / 能力 | 做什么 |
|-------------|--------|
| **Terminal** | 用本机终端跑命令 |
| **Browser** | 开浏览器、搜索、点击等 |
| **Delegate Task** | 孵化**子 Agent**（例如委派 **Claude Code CLI** 写代码 / 修 GitHub bug） |
| **Cron Job** | 定时任务，到点自动跑，无需人盯 |
| **Skill Management** | 创建 / 管理 Skills（程序记忆） |
| **MCP** | 可接 MCP（按用户问题选用） |

用完工具 → End Loop → 回复。下面是实测。

## 八、Harness + Loop 实测

### 8.1 Cron：每分钟宝可梦 / 开发者笑话（10 分钟）

**Desktop 指令示例：**  
「帮我设 Cron：接下来 10 分钟每分钟讲一个宝可梦笑话。」

**WhatsApp 同步试：**  
「告诉我一个开发者笑话……」（同样走到 Cron）

现象：

- 很快建成 Cron：`repeats 10 times / next 10 minutes`
- Terminal UI 上任务「挂起」时可能**不打印**，需再问「第一段笑话是什么」
- WhatsApp 侧提示：Joke machine 已启动；约一分钟后第一条
- 例：「为什么杰尼龟要穿越海洋？为了得到另一个潮汐。皮卡皮卡」
- 后来又收到：Java 开发者戴眼镜（can't see sharp）、UDP 笑话（不保证你收到）等
- **问题（作者吐槽给 Hermes 团队）：** 预期 10 个笑话，实际只主动推了约 2 个；除非主动问结果，Cron 似乎不继续更新——像 bug

### 8.2 Terminal：OS 版本 + Shell 最近 10 行

新会话提问：用终端查当前 OS，并读 Shell history 最后 10 行。

结果：

- 跑在本机 → 识别 **Mac OS 26.5**
- History 多为 `get status` 一类命令
- 流程：Agent Run → 调 Terminal → 取完 → **停循环** → 回复  
（Claude Code 用户早已熟悉；这里用来证明「Harness + Loop」日常就在用。）

### 8.3 Browser：YouTube 频道标题与播放量 → 新标题

任务：打开频道 **Sean's AI Stories**，看近作标题 + 观看数，为正在录的这期想新标题。

过程（系统设计视角，能力本身 Claude Code 也有）：

1. 调浏览器搜索  
2. URL 错 → 报错 → 换新 URL 再搜  
3. **自动写 Memory**：`.Hermes` → `memories` → **`memory.md`** 出现如 *YouTube scraping quirk*（踩坑后自我更新）  
4. 读到往期（含约 10 万播放）→ 给出新标题建议  
5. 中途还调过 Terminal 跑 Python、点击类工具  

Loop 故事线：

```
Gateway 提问 → Agent Run（Gemini Harness）
  → Browser（第一次失败）→ End 护栏：「没找到」
  → 用户/Agent 纠错 URL → 再 Browser + Terminal/Python
  → 成功 → End Loop（不再调工具）→ 答案
```

**剧透点：** Hermes 会**自己**把 Agent Memory 存进本地文件（见 Memory 专章）。

### 8.4 Delegate：两个并行子 Agent

指令：用 Delegate Task 孵化两个子 Agent——

1. 研究 **LM Eval Harness**  
2. 研究 **VLM 架构**  

现象：调用 `Delegate Task` → 两个后台 Agent **并行** → 汇总结果返回。

### 8.5 Delegate + Claude CLI：Hacker News Top 5 脚本（作者最爱）

指令：无头模式 spawn 使用 **Claude CLI** 的子 Agent；用 Claude 写 Python：拉 HN Top 5 → Markdown → 验证能跑。

流程：

```
Gateway 提问
  → Agent Run
  → Delegate Task
  → 子 Agent 跑 Claude Code（读过「Autonomous AI Agent / Claude」相关 Skill：如何 install、run、auth、委托编码）
  → 产出 fetch_hn.py
  → 主 Agent 再跑脚本
  → 返回 Top 5 结果（作者核对正确）
```

说明：这是完整 **Harness + Loop**；本轮**未**看到「自发学会新 Skill」，主要触发了 Memory 学习。显式追问后 Agent 也承认：技能是用户让建的，**还没在难任务后主动提议存 Skill**。作者期望：困难、迭代、**≥5 次工具调用**后应主动提议保存 Skill——当时任务还不够「难」。

### 8.6 显式 Memory / Skill 测试

| 操作 | 结果 |
|------|------|
| 「存到记忆：我最喜欢的测试框架是 Pytest」 | **显式**更新 `memory.md`，多出一块：favorite testing framework = Pytest（Gateway → 更新记忆工具） |
| 「过去会话里我对你说的第一句话是什么？」 | Loop 里用**搜索工具**；答：*What can you do Hermes? What makes you special?*（当时登录失败过；信息来自 **Episodic `state.db`** → Working Memory） |
| 「创建 Skill：`video prep`——口播英语、行内解释术语、不用 em dash、结尾固定句 *you can build anything you can learn anything*」 | **Skill Management** 工具 → `.Hermes/Skills/.../video-prep`；内容符合要求 |

### 8.7 Gateway 跨会话总结

当前聊天不知道其它会话细节时，让 Agent「总结今天做了什么」——应去拉 **state.db / memory.md / skills.md**。

它能概括到：即时笑话系统、查 OS、YouTube 数据、Hacker News 等 → 说明跨入口记忆在工作。

WhatsApp 与 Desktop：**同一 Hermes 后端**，Gateway 只是界面。想像挂在永不关机的虚拟机上：随时 WhatsApp 问「现在怎样」即可。

**设置提示：** 输入 `Hermes WhatsApp`，按说明配置即可。

## 九、Memory 三柱在 Hermes 里怎么落地（重点差异标红级）

### 9.1 总原则

- Skills + Memories **完全本地**，**不存云**  
- 每次 Agent Run 中/后：若发现错误该学、或任务重复 → **自改进环**摘要「以后服务用户该知道的新记忆」→ 写入两块文件型记忆  
- Working Memory 会吃进：Procedural + Semantic +（完整后）Episodic → Harness 更「优雅」

### 9.2 Procedural Memory（过程 / 程序记忆）

| 项 | Hermes 实现 |
|----|-------------|
| 存什么 | Agent「怎么做」：行为方式、可复用流程 |
| 路径 | `.Hermes` 下 **Skills**（如 `*skill*.md` / Hermes Skills 目录） |
| 例子 | **Autonomous AI Agent · Claude Code**：如何把编码委托给 Claude Code CLI（安装、运行、认证等） |
| 写入 | Skill Management 工具；理想上难任务后应**主动**提议存 Skill（演示里显式创建了 `video prep`） |

### 9.3 Semantic Memory（语义 / 耐久事实）

| 项 | Hermes 实现 |
|----|-------------|
| 存什么 | 耐久事实、用户画像、习惯（如 Pytest、YouTube scraping quirk） |
| 路径 | **`memory.md`**（memories 目录） |
| **关键差异** | **不用 Embedding / RAG**；**纯文本 + Top Keyword（关键词）** 检索。作者也不确定为何如此，但行为就是文本处理 |
| 写入 | 显式「save to memory」；或 Run 中自动（如爬虫翻车后写 quirk）；也可由情节巩固蒸馏而来 |

### 9.4 Episodic Memory（情节记忆）

| 项 | Hermes 实现 |
|----|-------------|
| 存什么 | 聊天记录、带日期事件 |
| 路径 | 本地 **`state.db`**（`.Hermes` 末尾可预览）——**不落云** |
| 召回 | 可为 RAG 或 SQL 思路注入 Working Memory；实测「第一句话」靠搜过去会话 |
| 每次 Run 后 | 信息流入 Episodic |
| 巩固 | 随时间用 **Hermes 辅助模型（更便宜、非主力）** 摘要聊天 → **蒸馏事实进 Semantic（`memory.md`）** |

### 9.5 自改进环（一张嘴说清）

```
Agent Run（工具循环）
    │
    ├─► 写 Episodic：state.db
    ├─► 错 / 重复 / 显式要求 → 更新 Skills（Procedural）
    ├─► 错 / 习惯 / 显式要求 → 更新 memory.md（Semantic）
    └─► 辅助小模型定期 consolidate Episodic → 再灌 Semantic
```

作者观感：图比通用版更复杂，但这是 **Hermes 工作方式的完整总结**——Harness 在**自改进、偏自治**状态下跑。

## 十、LLMOps / Eval：Hermes 缺哪一块

原图右侧蓝色 **LLMOps** 盒：Trace → Eval → Diagnose → Ship。

| 能力 | Hermes 现状（作者调研时） |
|------|---------------------------|
| **Trace / Logs** | 有：**trajectory export**、日志；`.Hermes` → Logs：错误、Agent runs、**Gateway logs**（入口是 WhatsApp 还是 Desktop） |
| **Eval** | **基本没有**；需自建 |
| 通用 Eval 栈对照 | LangSmith、LangFuse 等：整次 Run、工具调用次数、LLM 调用次数、是否用便宜模型做 Semantic 摘要……Hermes 不自带这套 |

作者猜测：本地系统，刻意留给用户定制。  
结论：**能追踪，不能开箱评测**；LLMOps 闭环要自己接。

## 十一、和 Claude Code / OpenClaw 比什么

| 维度 | 共同点 | Hermes 相对特点 |
|------|--------|-----------------|
| 能力面 | 终端、浏览器、循环、子 Agent、Cron、Skill……日常都在用 | — |
| 存储 | 都可有记忆/技能 | **本地自更新** Skills + Memory + `state.db`；隐私向（作者：不明白为何不能上云，但本地是卖点） |
| 人格 / System | Claude 有 Customize instruction | **`SOUL.md`** |
| Semantic 检索 | 常见 Embedding/RAG | **纯文本关键词** |
| 自改进 | Claude 多需人驱动做 Skill | **内置学习环**（从经验写 Skill/Memory）；演示里 Memory 自动更明显，Skill 自发仍偏弱 |
| Gateway | — | WhatsApp / Desktop，类 OpenClaw |
| LLMOps | 生态工具多 | **无完整 Eval**；有 logs/trajectory |

作者判断：实现上是**相当标准的 Agent Harness**，工程量肯定不小，但概念不炫技——适合当「今天该怎么做产品」的范本：

> 每个 AI Agent 工具都应能 **self-improving / self-evolving**；本地或云端按需求选。  
> **Skill + Memory 积累**可能是最值钱的部分——用户被「记得我是谁」锁住（作者自己也不换模型了：Claude 里有自己、公司 Automanus.io、做 YouTube 的风格等 Semantic）。

## 十四、面试 / 对外口述版

Hermes（Nous Research）是本地开源 Agent Harness：Gateway（Desktop/WhatsApp）进 Working Memory，LLM 在 Loop 里调终端、浏览器、Cron、MCP、Delegate（含 Claude CLI），用护栏结束再回复。

记忆三柱本地化——Skills 做 Procedural，`memory.md` 做 Semantic（**关键词纯文本，非 RAG**），`state.db` 做 Episodic，并用便宜辅助模型往 Semantic 蒸馏；Run 中还能因错误/重复**自写记忆与技能**。

相对 Claude Code：更强调本地自改进与隐私；相对完整 LLMOps：有 trajectory/logs，**缺 Eval**，要自建。

价值不在概念新，而在把「标准 Harness + 自进化 Memory/Skills」做成可下载产品；长期护城河是**越用越懂你的技能与记忆积累**。
