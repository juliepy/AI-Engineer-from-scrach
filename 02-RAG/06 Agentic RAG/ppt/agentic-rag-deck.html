02
回顾

你现在的系统:Naive RAG

Phase 2 你构建的 RAG 是 "Naive RAG"——每次用户提问都走同样的固定流程,一条道走到黑:

用户问题→ Embedding→ 向量搜索 top-5→ 拼入 Prompt→ LLM 生成回答
📌 特点:无脑、固定、可预测。查询模式单一时(比如只查 FAQ)它够用;但一旦问题变复杂,这条直线就撑不住了——见下一页。
03
Naive RAG 的局限

固定流程的三个硬伤

① 不需要也检索

用户说"你好",系统也去向量库搜一遍——浪费 token 和时间。

② 一次不够没法再搜

问"对比 RAG 和 Fine-Tuning 优缺点",一次 top-5 可能只覆盖 RAG,另一半文档没搜到。

③ 不好也无法自我修正

搜到的文档不相关,LLM 只能硬着头皮编答案,越编越离谱。

🔍 根因:Naive RAG 没有"判断力"——它不会想"这题要不要搜""搜的够不够""搜错了怎么办"。缺的正是一个会决策的大脑。
04
Agentic RAG

= 给 RAG 加一个"大脑"

在 RAG 前面加一个 Agent 层,让它自主决策:要不要检索、检索什么、检索几次、结果够不够、要不要换个库再搜。

用户问题→ 🧠 Agent
  • 是否检索? 简单寒暄直接答,复杂问题才动向量库。
  • 检索什么? 可以改写 query、拆成多个子问题分别搜。
  • 检索几次? 一次不够就补搜,换库再搜,直到信息够。
  • 够不够? 检索结果不足就承认"没找到",不编造。
🧠 一句话:Agentic RAG = Agent + RAG。把"固定流水线"换成"会临场决策的助手"。
05
全维度对照

Naive vs Agentic:差在哪

维度Naive RAGAgentic RAG
检索时机每次都检索Agent 判断是否需要检索
检索策略固定 query、固定 top-k可改写 query、调 top-k、多次检索
回答策略一次生成可验证回答、不足时补充检索
工具只有向量搜索向量 + 网页 + SQL + 计算器
错误处理无失败 fallback、不确定拒答
⚖️ 本质:Naive 是写死的流水线,Agentic 是带判断力的决策循环。多一个"大脑",就多了应对复杂查询的能力。
06
该不该升级

别盲目上 Agentic RAG

Agentic 更强,但也更慢、更贵、更难调试。升不升级,看你的查询模式有多样。

✅ 值得升级
  • 查询模式多样:有时要文档、有时要搜网页、有时不用检索
  • 问题常需多步 / 多库组合信息
  • 对回答质量、可信度要求高
❌ 没必要(Naive 就够)
  • 查询模式单一:比如只查 FAQ
  • 对延迟/成本极敏感
  • 问题都能一次 top-k 覆盖
📌 判断口诀:查询模式越多样,Agentic 的价值越大;模式越单一,越该省心用 Naive。
07
实现架构

Agent 决策层 + 多工具

核心是一个决策循环:Agent 拿到问题,在一堆工具里自主选择、多轮调用,信息够了才回答。

🧠 Agent 决策层

判断用不用工具、用哪个、够不够,最多 N 轮。

🔧 工具箱

search_docs / search_web / SQL / calculator。

📚 多向量库

不同知识域各一个库,Agent 决定搜哪个。

📊 LangSmith

全程追踪每轮决策、检索、延迟与 token。

🏗 记住这个结构:一个大脑 + 一个工具箱 + 多个知识库 + 一路留痕。后面所有代码都在填充这四块。
08
实现 · 决策循环

核心就是一个带工具的循环

def agentic_rag(question): tools = [search_docs, search_web, calculator] # Agent 的三个工具 messages = [{"role":"system", "content": SYSTEM_PROMPT}, {"role":"user", "content": question}] for _ in range(5): # 最多 5 轮决策(停止条件) response = llm(messages, tools=tools) if response.tool_calls: # Agent 决定调工具 for call in response.tool_calls: result = execute(call) messages.append(tool_result(call.id, result)) else: return response.content # 不需更多检索 → 直接回答 return "无法回答,请换个问题试试" # 触发上限 → 降级
🔁 看点:有 tool_calls 就执行并回填,没有就直接回答。Agent 自己决定循环几轮——简单问题 1 轮,复杂问题多轮。
09
Agent 的路由决策

System Prompt 就是它的"行动纲领"

Agentic RAG 的灵魂在 System Prompt——明确告诉 Agent 什么时候用什么工具:

你是一个知识助手。根据用户问题,选择合适的策略: 1. 可以直接回答(如"你好""谢谢") → 直接回答,不调用工具 2. 需要查找内部文档 → 调用 search_docs 3. 内部文档不够 → 补充调用 search_web 4. 需要计算 → 调用 calculator 5. 检索内容无法回答 → 说"我在文档中没有找到相关信息" 重要:不要编造信息。回答必须基于检索到的内容。
🧭 路由质量 = 提示词质量。把"何时用/何时不用/找不到怎么说"写清楚,Agent 才不会乱搜、瞎编。这是本节最该反复打磨的一段。
10
工具箱

从"只有向量搜索"到一整套

search_docs
向量搜索

检索内部文档,Agentic RAG 的主力工具。

search_web
网页搜索

内部文档不够时补充,覆盖最新/外部信息。

SQL 查询
结构化数据

精确数字、报表类问题走数据库而非向量。

calculator
数值计算

百分比、汇总等,别让 LLM 硬算。

🔧 工具越多,Agent 越能"对症下药"——但每个工具都要单一职责、结构化返回、有清晰 description,否则 Agent 会选错。
11
进阶 · 多库路由

三个向量库,搜哪个?

生产里往往不止一个向量库:产品文档一个、政策合规一个、历史工单一个。Agent 要先判断"这题属于哪个域",再去对应的库搜。

按知识域分库

产品 / 政策 / 工单各一个库,互不污染,检索更准。

Agent 先分类再路由

先判断问题归属,再只搜相关库;跨域问题就搜多个再合并。

合并 & 去重

多库结果按相关性得分合并、去重,再交给 LLM 生成。

📚 这正是本节现场演示的重头戏:看 Agent 怎么在三个向量库之间做决策——该搜哪个、要不要跨库、结果怎么合。
12
进阶 · 检索自我修正

一次搜不好,就改写再搜

问题:一次检索覆盖不全
  • "对比 A 和 B" 一次 top-5 常只搜到一边
  • 用户用词和文档用词对不上
  • 问题太笼统,搜出来一堆噪声
Agent 的自我修正
  • 拆子问题:"搜 A" + "搜 B" 分别检索
  • 改写 query:换更贴文档的措辞重搜
  • 多次检索:结果不足就补搜(限重试次数)
💡 关键:改写/补搜要有次数上限(如最多 2 次),否则一道题能把 Agent 卡在检索循环里,又慢又烧钱。
13
错误恢复 & 降级

检索失败,怎么优雅收场

故障场景恢复 / 降级策略
检索结果不相关Agent 改写 query 重新搜索(最多重试 2 次)
检索服务超时降级到直接用 LLM 回答,并标注"未经文档验证"
回答不确定 / 信息不足明确拒答:"文档中没有足够信息回答这个问题"
🛟 一条原则:宁可承认"没找到",也不要编造。 拒答不是失败,乱编才是——尤其在生产环境,可信度比"看起来什么都会答"重要得多。
14
可观测性 · LangSmith

复杂调用链,靠 Trace 看清

Agentic RAG 的调用链比 Naive 复杂得多。LangSmith Trace 让你把黑盒变透明:

  • Agent 做了几轮决策
  • 每轮调了什么工具、传了什么参数
  • 检索返回了哪些文档、相关性得分多少
  • 最终回答基于哪些检索结果生成
  • 每步的延迟和 token 消耗
📊 没有 Trace,Agentic RAG 出问题你根本不知道是"没搜到""搜错库"还是"搜到了但没用"。可观测性是 Agentic RAG 的标配,不是加分项。

18
常见问题

现场最常被问的两个

Q:比 Naive 慢多少?

取决于 Agent 做了几轮决策。简单问题(不检索)反而更快;复杂问题(多次检索)可能慢 2–3 倍。实测平均 2.1s → 3.5s,但 Faithfulness 从 0.78 → 0.91。

Q:什么时候不该用?

查询模式单一(比如只查 FAQ)时,Naive 就够了。Agentic 的价值在查询模式多样——有时要文档、有时要搜网页、有时不用检索。

🧊 补充:别为了"显得高级"上 Agentic。先测你的真实查询分布:如果 90% 都是同一类简单查询,Naive + 少量规则可能更划算。

小结

带走这三句话

  • Agentic RAG = 给 RAG 加一个大脑:Agent 决定何时检索、检索什么、检索几次、怎么组合信息。
  • 代价与收益:代价是更高的延迟和成本,收益是显著更好的回答质量(Faithfulness 0.78→0.91)和用户体验。
  • 生产四要素:清晰的路由 Prompt、多工具/多库、错误恢复与降级、LangSmith 全链路可观测——缺一不可。
